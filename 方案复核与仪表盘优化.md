# API 价格追踪项目 — GitHub 方案复核 & 仪表盘优化方案

> 日期：2026-06-25　|　复核对象：现有 GitHub Actions 自动化方案 + 现有 ECharts 仪表盘
> 决策前提：托管改用 **GitHub Pages**，仪表盘采用 **现代浅色商务风**

---

## 第一部分：GitHub 方案可行性复核

### 1.1 总体结论

**方案整体可行，链路设计是成立的，但当前这一版有 3 个必须修的硬伤 + 几个建议改进点。**

核心判断：抓数（`fetch_prices.py`）和生成（`visualize.py`）这两步本身没问题，真正的断点全部集中在「最后一步部署」和「仓库可见性」上。把这几处改掉，就能得到一条电脑关机也能跑、零腾讯云依赖、全免费的自动化链路。

### 1.2 现有链路梳理

```
cron 触发 (UTC 1:00 = 北京 9:00)
  └─ checkout 代码（含历史 prices.xlsx）
     └─ 装 Python 3.13 + Node 20 + EdgeOne CLI
        └─ fetch_prices.py：拉 tokenpricing → 全量入 SQLite + 12 旗舰入 Excel
           └─ visualize.py：读 Excel → 生成 price_dashboard.html
              └─ 拷到 deploy_dist/index.html
                 └─ EdgeOne CLI 部署    ← ❌ 当前断点
                    └─ commit prices.xlsx 回仓库
```

### 1.3 确认没问题的部分（可放心保留）

- **cron 时间换算正确**：`0 1 * * *` 是 UTC 1:00，对应北京时间 9:00，脚本里用 `datetime.now()` 取日期在 UTC 环境下也是同一天，没有跨日错位。
- **权限正确**：`permissions: contents: write` 足以让 Actions 把数据 commit 回仓库。
- **数据源稳健**：tokenpricing 走 `raw.githubusercontent.com`，零 key 零限流、GitHub CDN 兜底，几乎不宕机，价格单位已是 per-million 无需换算。
- **免费额度充足**：此任务每天约 2–3 分钟，每月约 90 分钟。即便用私有仓库（每月 2000 分钟额度）也绰绰有余；改 Public 后 Actions 完全免费无上限。
- **存储策略合理**：`prices_full.db` 每次重建不入仓库，只 commit 体积小的 `prices.xlsx` 保留旗舰历史趋势——这个取舍是对的。

### 1.4 硬伤（必须修，否则方案跑不通）

**硬伤 1 — 部署目标 EdgeOne 整条卡死**
EdgeOne 同时踩了两个雷：Zone 处于 `init` 未激活状态 + 腾讯云账号欠费导致服务冻结。结果是 `fetch` 和 `visualize` 都能成功，但工作流最后的 `edgeone makers deploy` 必然报 `Zone is not active` 而失败，整个 run 标红。**这是当前方案最致命的断点。** → 既然你已决定绕开，改用 GitHub Pages 即可彻底摆脱腾讯云。

**硬伤 2 — 私有仓库与 GitHub Pages 互斥**
迁移指南和记忆里写的是"仓库选 Private（私有，因为含数据）"。但 **GitHub Pages 对私有仓库是付费功能**（需 GitHub Pro/Team/Enterprise）；免费账号下私有仓库无法开启 Pages。
解决办法二选一：
- **推荐：仓库设为 Public。** 这个项目追踪的是各家厂商的公开 API 报价，数据本身没有任何商业机密或个人隐私，公开完全没问题，且省钱省心。
- 备选：保留私有 + 升级 GitHub Pro（约 $4/月），与"零月费"目标相悖，不推荐。

**硬伤 3 — `.gitignore` 把生成的 HTML 忽略了**
当前 `.gitignore` 第 11–12 行忽略了 `price_dashboard.html` 和 `index.html`。改用 Pages 后，部署的就是这个 HTML，需要让 Actions 能拿到它。
推荐做法：**不**把 HTML 提交进仓库（保持忽略，仓库历史干净），而是在工作流里把它放进一个临时 `_site/` 目录，用 GitHub 官方的 `upload-pages-artifact` 上传部署。这样 HTML 始终是"每次构建产物"，不污染 Git 历史。

### 1.5 建议改进点（非阻塞，但值得做）

- **删掉无用步骤**：改 Pages 后，工作流里的"装 Node""装 EdgeOne CLI""EdgeOne 部署"三步全部删除，`EDGEONE_API_TOKEN` secret 也不再需要，每次 run 能省 30–60 秒。
- **加失败通知**：目前 run 失败你不会收到任何提醒。建议在 Settings → Notifications 打开 Actions 失败邮件，或在工作流末尾加一步失败时自动开 issue。
- **加数据新鲜度校验**：tokenpricing 万一停更，价格会"静默不变"而你不知道。建议在 `fetch_prices.py` 里校验 `generated_at`，如果超过比如 3 天没更新就在日志里告警。
- **趋势数据需要时间积累**：目前只有 6/23、6/25 两天数据，趋势图意义有限。这是时间问题不是 bug，正常每日跑积累即可；可在图上标注"数据积累中"。

### 1.6 改造后的目标架构

```
GitHub Actions (cron 每日 UTC 1:00)
  ├─ fetch_prices.py   → 全量入 SQLite + 12 旗舰入 Excel
  ├─ visualize.py      → 生成 index.html
  ├─ commit prices.xlsx 回仓库（版本化历史）
  └─ 部署到 GitHub Pages（官方 artifact 方式）

访问地址：https://<用户名>.github.io/<仓库名>/
特点：全免费、零腾讯云、电脑关机也跑、数据 Git 版本化
```

### 1.7 GitHub Pages 改造操作清单

1. **仓库设为 Public**（Settings → General → Danger Zone → Change visibility）。
2. **替换工作流**为下方 1.8 的新版 YAML。
3. **开启 Pages**：Settings → Pages → Source 选 **GitHub Actions**。
4. **删除** 不再需要的 `EDGEONE_API_TOKEN` secret 和 EdgeOne 相关本地文件引用。
5. 首次 **手动触发**（Actions → Run workflow）验证，约 2 分钟后访问 `https://<用户名>.github.io/<仓库名>/`。
6. 验证成功后，自定义域名 `apipricetracing.gf-dsai.com` 可选：在 Pages 设置里绑定 CNAME 指向 `<用户名>.github.io`（这一步与 EdgeOne 无关，DNS 干净）。

### 1.8 新版工作流 YAML（可直接替换 `.github/workflows/daily-update.yml`）

```yaml
name: Daily API Price Update

on:
  schedule:
    - cron: '0 1 * * *'   # 每日 UTC 1:00 = 北京 9:00
  workflow_dispatch:

permissions:
  contents: write   # commit 数据回仓库
  pages: write      # 部署 Pages
  id-token: write   # Pages 部署鉴权

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  update:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.13'

      - name: Install deps
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Fetch prices
        run: python fetch_prices.py

      - name: Visualize dashboard
        run: python visualize.py

      - name: Commit data back to repo
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add prices.xlsx
          git diff --staged --quiet || git commit -m "data: update $(date -u +%Y-%m-%d) price snapshot"
          git push

      - name: Prepare site
        run: |
          mkdir -p _site
          cp price_dashboard.html _site/index.html

      - name: Configure Pages
        uses: actions/configure-pages@v5

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: _site

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

> 注意：相比旧版，删除了 Setup Node、Install EdgeOne CLI、EdgeOne 部署三步，新增了 Pages 三步。`prices_full.db` 仍不入仓库。

## 第二部分：大模型 API 价格追踪仪表盘 — 优化方案

### 2.1 现有仪表盘问题诊断

逐项看 `visualize.py` 生成的页面，主要问题：

- **配色割裂**：顶部是深色渐变 header（`#1a1a2e`），下面却是浅灰 body，风格不统一，像两个页面拼起来的。
- **信息密度低、要一直滚**：4 张图纯纵向堆叠，每张 420–520px 高，一屏看不到两张，"观察趋势"要往下翻很久。
- **核心目的被埋没**：项目目的是"观察价格趋势"，但趋势折线图排在第 3 位，最该突出的图被压在下面。
- **交互是半成品**：趋势图代码里定义了厂商筛选 `buttons`，但前端根本没接上，无法按厂商筛选。
- **KPI 卡片偏弱**：只是左边一条彩色边的白卡，没有"涨跌"信号——追踪价格变动，最该一眼看到的就是"谁涨了谁降了"。
- **标题层级混乱**：每张图用 ECharts 内置 title，和页面大标题混在一起，没有清晰的视觉层级。
- **移动端未适配**：固定宽度图表，手机上挤成一团。

### 2.2 新设计目标（现代浅色商务风）

- **统一浅色基调**：`#F7F8FA` 页面底色、纯白卡片、柔和阴影（`0 1px 3px rgba(0,0,0,0.06)`）、12px 圆角，干净适合日常查看。
- **趋势优先**：价格趋势折线图放大、置顶为"主图"，这是项目的核心价值。
- **一屏概览**：顶部 KPI 行 + 主趋势图组成第一屏，其余图用响应式网格排在下方。
- **加涨跌信号**：KPI 和价格表里显示 vs 上一快照的涨跌幅，涨红降绿（国内习惯），一眼看到变动。
- **真交互**：厂商筛选 chips、旗舰/次旗舰切换、输入价/输出价切换，全部前端 JS 完成，不依赖后端。
- **响应式**：桌面多列网格，手机自动单列。
- **保留品牌色**：各厂商主色（OpenAI 绿、Anthropic 橙等）保留用于区分，但统一到柔和色板，避免刺眼。

### 2.3 新页面结构（线框）

```
┌─────────────────────────────────────────────┐
│ 顶栏（浅色）                                   │
│ 大模型 API 价格追踪    [数据源徽章] 更新于 …  │
├─────────────────────────────────────────────┤
│ ① KPI 行（4 卡，每卡带涨跌角标）              │
│  跟踪厂商 | 跟踪模型 | 数据跨度 | 最便宜旗舰  │
├─────────────────────────────────────────────┤
│ ② 主图：价格趋势折线（大）                    │
│    [全部] [OpenAI] [Anthropic]… 厂商 chips    │
│    [输入价/输出价 切换]  [旗舰/次旗舰 切换]    │
├──────────────────────┬──────────────────────┤
│ ③ 旗舰 vs 次旗舰对比   │ ④ 输入/输出价格散点   │
├──────────────────────┴──────────────────────┤
│ ⑤ 综合价格排行（横向条）                      │
├─────────────────────────────────────────────┤
│ ⑥ 最新价格明细表（可排序，带涨跌列）          │
└─────────────────────────────────────────────┘
```

### 2.4 逐图改进点

- **① KPI**：保留 4 卡，但"最便宜旗舰"等数值旁加涨跌角标（▲ +5% 红 / ▼ −3% 绿）；卡片改为顶部细色条 + 大数字 + 小注释的层级。
- **② 趋势图（主图）**：高度提到 ~460px 置顶；接上厂商筛选 chips；加输入/输出价切换；只有 1 天数据时显示友好占位文案（现有逻辑保留）。
- **③ 旗舰 vs 次旗舰**：分组柱状图保留，配色统一柔化，数值标签更克制。
- **④ 散点图**：输入价 vs 输出价分布，旗舰用实心圆、次旗舰用菱形（保留），图例改为可点击筛选。
- **⑤ 价格排行**：横向条形，按综合价（输入+输出）排序，旗舰/次旗舰双色区分。
- **⑥ 新增明细表**：原生可排序表格，列含厂商/模型/定位/输入价/输出价/较上次涨跌，方便精确查数和复制。

### 2.5 技术实现要点

- **仍用 ECharts CDN，保持单文件**：目标 HTML 控制在 ~25–35KB，零本地 JS 依赖，浏览器直接渲染不下载（吸取之前 Plotly 4.67MB 强制下载的教训）。
- **数据一次性注入**：`visualize.py` 把 `prices.xlsx` 的全部数据序列化成一个 JSON 塞进页面全局变量，所有图表和筛选切换都在前端基于这份数据计算——筛选/切换不需要重新生成页面。
- **涨跌计算**：前端对每个模型取"最新快照 vs 上一快照"算变化率，驱动 KPI 角标和表格涨跌列。
- **响应式**：用 CSS Grid + `@media` 断点，桌面 2 列、手机 1 列；图表监听 `resize` 自动重绘（现有逻辑保留扩展）。

### 2.6 可选增强（后续迭代）

- **"近期变动"提示条**：页面顶部自动列出最近有价格变化的模型，突出"变动"本身。
- **用上全量数据**：目前只展示 12 旗舰，DB 里 3200+ 模型未利用。后续可加一个"全部模型搜索/筛选"页，按需查询任意模型价格。
- **自定义域名**：Pages 稳定后绑定 `apipricetracing.gf-dsai.com`，CNAME 指向 `<用户名>.github.io`，DNS 链路干净，不再依赖 EdgeOne。

### 2.7 落地步骤（分阶段）

- **P0｜先让它自动在线**：改 `.gitignore`、换工作流、仓库转 Public、开 Pages（第一部分内容）。这一步做完，电脑关机也能每日更新在线仪表盘。
- **P1｜重做视觉**：重写 `visualize.py` 的 `build_html`，落地浅色商务风 + 趋势优先布局 + KPI 涨跌。
- **P2｜加交互**：厂商筛选 chips、输入/输出切换、可排序明细表、近期变动提示条。

## 一句话总结

GitHub 自动化方案的骨架是对的，卡点全在"部署到 EdgeOne"这一环——绕开它改用 GitHub Pages（仓库转 Public + 官方 Pages artifact 部署）即可跑通，且彻底零腾讯云、零月费。仪表盘则建议推倒重做：统一浅色商务风、把价格趋势图提为置顶主图、KPI 加涨跌信号、补上厂商筛选与可排序明细表。
